# ПЛАН МОДУЛЯ ОПЕРАЦИОННОГО УПРАВЛЕНИЯ (TaskOps)

**Проект:** eMedosmotr-map
**Версия:** 1.0
**Дата:** 2026-05-08
**Статус:** Проектирование

---

## 0. TL;DR

Встроенный в eMedosmotr-map модуль управления задачами и проектами для команды компании и **внешних разработчиков на аутсорсе**. Аналитики ставят задачи, руководители видят прогресс на красивых дашбордах и таймлайнах, разработчики работают в едином пространстве с минимально необходимым интерфейсом.

**Ключевые принципы:**
- **Отдельное пространство** — модуль не «вкладка карты», а самостоятельный режим. При переходе в «Задачи» карта отключается полностью (см. §4.1.1).
- **Адаптивность с двумя сценариями** — desktop для аналитиков/руководителей, mobile для инженеров и разработчиков на ходу (см. §4.5).
- **Linear-style минимализм** — никаких 200 полей, как в Jira.
- **Notion-style гибкость** — несколько views (List, Board, Timeline, Calendar) поверх одних и тех же данных.
- **ClickUp-style дашборды** — но без визуального шума.
- **Asana-style таймлайны** — Gantt с зависимостями, drag-n-drop.
- **Изолированный аутсорс** — внешние разработчики видят только свои задачи и проекты.

## 1. БЕНЧМАРК: ЧТО БЕРЁМ И ЧТО ОТБРАСЫВАЕМ

### 1.1 Сравнение лидеров

| Продукт   | Сильные стороны (берём)                                           | Что НЕ берём                                              |
|-----------|--------------------------------------------------------------------|------------------------------------------------------------|
| **Linear**    | Скорость, ⌘K-палитра, keyboard-first, Cycles (спринты), статусы с группами, opinionated UX | Слишком жёсткая модель issues — нам нужны проекты внедрения |
| **Jira**      | Зависимости задач, sprint-burndown, JQL-фильтры, roadmap         | Перегруженность, медленный UI, сложная конфигурация        |
| **ClickUp**   | Multi-view (List/Board/Gantt/Calendar/Timeline), кастом-поля, дашборды-конструктор | Feature creep, 15 кликов до простого действия              |
| **Notion**    | Database views, inline-документы, шаблоны, ссылки между сущностями | Слабый task-management (нет нормальных зависимостей, бэклога) |
| **Asana**     | Timeline (Gantt), Goals→Projects→Tasks, My Tasks inbox, milestones | Перегруженные карточки, дорогие планы                      |

### 1.2 Лучшие практики, которые забираем

1. **Иерархия объектов:** `Goal → Project → Task → Subtask` (Asana-style, не глубже 2 уровней вложенности задач).
2. **Статусы как фазы**, не как чекбоксы: `Backlog → To Do → In Progress → In Review → Done → Cancelled` (Linear).
3. **Cycles / Sprints** — двухнедельные итерации с автоматическим переносом незавершённых задач.
4. **Inbox / My Tasks** — единая входящая для каждого пользователя (Asana, Linear).
5. **⌘K command palette** — быстрое создание/поиск/навигация (Linear).
6. **Ярлыки и приоритеты:** P0–P3 + цветные labels (Linear, GitHub).
7. **Зависимости:** `blocks / is blocked by / relates to` (Jira), визуализируются на Gantt.
8. **Timeline (Gantt) с drag-n-drop** дат и связей (Asana, ClickUp).
9. **Дашборды-виджеты:** burndown, throughput, lead time, by-assignee, by-status (ClickUp + Linear Insights).
10. **Activity feed + @mentions + комментарии в Markdown** (везде).
11. **Шаблоны проектов и задач** (Notion, ClickUp) — для типовых сценариев «внедрение в области».
12. **Real-time co-editing статусов и комментариев** через WebSocket (Linear).
13. **Связь задач с реальными объектами карты** — наша уникальная фича: задача может быть привязана к военкомату/ЦРБ/региону.

### 1.3 Чего сознательно НЕ делаем

- ❌ Time tracking с таймером (Toggl-style) — для аутсорса не нужно, отчётность по часам идёт вне системы.
- ❌ Формы/опросы/CRM — это не наш модуль.
- ❌ Глубокие automations (if-this-then-that) на старте — в v2.
- ❌ Чат внутри задач отдельно от комментариев — не плодим коммуникационных каналов.
- ❌ Множественные workspace — у нас одна компания.

## 2. ПРИВЯЗКА К НАШЕЙ ДЕЯТЕЛЬНОСТИ

### 2.1 Кто и зачем будет пользоваться

| Роль                       | Что делает в модуле                                                                |
|----------------------------|------------------------------------------------------------------------------------|
| **Руководитель**           | Смотрит дашборды (по областям, по подрядчикам, по релизам), утверждает приоритеты. |
| **Аналитик (наш)**         | Ставит задачи разработчикам-аутсорсерам, ведёт бэклог, пишет требования.           |
| **Региональный инженер**   | Получает оперативные задачи (выехать, настроить, обучить), отчитывается фото/комментарием. |
| **Разработчик-аутсорс**    | Видит только свои проекты, берёт задачи из спринта, меняет статус, комментирует.   |
| **Аналитик аутсорсера**    | Может декомпозировать задачи (с правами writer в выделенных проектах).             |

### 2.2 Типовые сценарии

1. **Внедрение в новой области** — проект-шаблон на 30+ задач (договор → выезд → установка ПО на N компьютеров → обучение → подключение оборудования → передача в сопровождение). Срок 4–8 недель, Gantt с зависимостями.
2. **Доработка ИС eMedosmotr** — продуктовый бэклог, спринты по 2 недели, аналитики компании ставят задачи аутсорсу.
3. **Инциденты на местах** — задача рождается из карты («Военкомат X не передаёт данные третий день») с автопривязкой к объекту.
4. **Призывная кампания** — большая Goal-цель с десятком проектов под ней, дашборд для руководства в реальном времени.

### 2.3 Уникальные фичи под наш домен

- **Привязка задачи к объекту карты** (`location_id`): при клике на маркер видна история задач по этому военкомату/ЦРБ.
- **Автогенерация задач из правил**: «если объект не передаёт данные >3 дней → создать задачу регионального инженера». (Celery-job + правила в админке.)
- **Шаблон проекта «Внедрение в области»** с предзаполненной структурой задач и зависимостей.
- **Двухконтурный доступ:** аутсорсеры видят только проекты с тегом `external` и только назначенные на них задачи.

## 3. МОДЕЛЬ ДАННЫХ

```
Project (проект внедрения / продуктовый эпик)           ← v1
  ├── estimate_type: t_shirt | hours                      ← выбирается на проекте
  ├── Cycle (двухнедельный спринт; опц.)
  └── Task
        ├── Subtask
        ├── estimate: XS/S/M/L/XL | число часов (в зависимости от estimate_type проекта)
        ├── Comment
        ├── Attachment
        ├── Dependency (blocks / blocked_by / relates_to)
        ├── Label (many-to-many)
        ├── Assignee (один основной + watchers)
        └── LocationLink (опц., FK на locations нашей карты)

Goal (стратегическая цель)                               ← v1.1
  └── Project
```

### 3.1 Ключевые поля Task

```
id, project_id, cycle_id?, parent_task_id?
title (string, обязательное, до 200 символов)
description (markdown)
status (enum: backlog | todo | in_progress | in_review | done | cancelled)
priority (enum: p0_urgent | p1_high | p2_medium | p3_low)
assignee_id (FK users)
reporter_id (FK users)
labels (M2M)
estimate (опц.: t-shirt XS/S/M/L/XL ИЛИ часы)
start_date, due_date (для Gantt)
location_id (опц., FK locations)
is_external_visible (bool, для аутсорса)
created_at, updated_at, completed_at
position (для drag-n-drop сортировки в колонках)
```

### 3.2 Права доступа (RBAC + scope)

| Роль          | Goals | Projects                | Tasks (внутри доступного проекта) |
|---------------|-------|-------------------------|------------------------------------|
| admin         | CRUD  | CRUD                    | CRUD                               |
| manager       | R     | CRUD назначенных        | CRUD                               |
| analyst       | R     | RW назначенных          | CRUD                               |
| engineer      | R     | R назначенных           | RW своих, R остальных              |
| external_dev  | —     | R только `external=true` и где есть membership | RW назначенных, R соседних       |

Каждый Project имеет таблицу `project_members(user_id, role)` — это финальная граница видимости для аутсорсеров.

## 4. UX И ИНТЕРФЕЙС

### 4.1 Принципы дизайна

- **Контент важнее хрома** — минимум границ, максимум воздуха. Палитра уже в `theme.ts`.
- **Один primary action на экране** — кнопка «New Task» всегда в правом верхнем.
- **Keyboard-first**: ⌘K, `c` — create, `g p` — go to projects, `1–4` — приоритет, `s` — статус.
- **Тёмная тема из коробки** (у нас уже есть в проекте).
- **Mobile-first для исполнителя, desktop-first для руководителя/аналитика** — см. §4.5.

### 4.1.1 Модуль как отдельное пространство (Workspace switch)

Модуль операционного управления — **самостоятельный режим приложения**, а не вкладка внутри карты.

- В шапке слева — переключатель пространств: **«Карта»** ↔ **«Задачи»** (по аналогии с переключением workspace в Linear/Notion). Опционально — `⌘.` для быстрого тогла.
- При переходе в «Задачи» **карта полностью отключается**: не загружается Yandex Maps SDK, не идут запросы геоданных, не висят WebSocket-каналы карты, не работают её хоткеи. Это даёт быструю загрузку и чистый контекст.
- При переходе обратно в «Карту» — наоборот, выгружается состояние TaskOps (кроме лёгкого in-memory кэша React Query).
- Технически: **lazy-loaded route bundles** (`React.lazy` + `Suspense`), отдельные Zustand-сторы, отдельные WS-неймспейсы (`/ws/map`, `/ws/taskops`). При смене пространства — `disconnect()` старого сокета.
- Глубокие ссылки сохраняются: `/map/locations/123` и `/tasks/inbox` — независимые URL-пространства.
- **Кросс-связь сохраняется через ссылки, а не через совмещённый UI:**
  - В карточке объекта на карте — кнопка «Открыть задачи объекта» → переход в TaskOps с предфильтром по `location_id`.
  - В задаче с привязкой к объекту — кнопка «Показать на карте» → переход в режим «Карта» с центрированием.
- Состояние «где я сейчас» хранится в URL и в localStorage (`activeWorkspace`), чтобы при перезагрузке возвращать пользователя в нужный режим.

### 4.5 Адаптивность: ПК и мобильный

Один и тот же бэкенд, два продуманных layout'а. **Mobile — это не урезанная версия desktop'а, а отдельный flow**.

#### Desktop (≥ 1280px) — для аналитиков и руководителей
- Трёхколоночный layout: **сайдбар (240px) | основной контент | side-panel задачи (440px)**.
- Сайдбар: переключатель Карта/Задачи сверху, ниже — Inbox, Goals, Projects (древовидно), Cycles, Dashboard.
- Все views доступны: List / Board / **Timeline (Gantt)** / Calendar.
- Task detail открывается как **side-panel справа** (Linear-style), а не модальным окном — можно работать со списком и задачей одновременно.
- ⌘K, drag-n-drop, multi-select (Shift+click), bulk-actions.
- Полноразмерный дашборд руководителя (см. §4.4).

#### Tablet (768–1279px)
- Сайдбар сворачивается в иконочный (60px), разворачивается по hover/tap.
- Side-panel задачи занимает 50% ширины поверх контента.
- Gantt доступен, но с упрощённой плотностью.

#### Mobile (< 768px) — для региональных инженеров и разработчиков «на ходу»
- **Bottom tab bar** (нативный паттерн): `Inbox` · `Projects` · `+` (создать) · `Notifications` · `Profile`. Переключатель Карта/Задачи — в Profile или жестом swipe.
- Доступные views: **List** и **Board** (горизонтальный swipe между колонками). **Gantt и Calendar — спрятаны**, вместо них кнопка «Открыть на ПК для таймлайна».
- Task detail — **полноэкранная страница**, не панель. Возврат — стрелка назад / свайп от левого края.
- Действия по свайпу на карточке задачи: вправо — Done, влево — меню (отложить, сменить статус, назначить).
- Создание задачи — **bottom sheet** с минимумом полей (title, project, assignee, due_date), остальное редактируется потом.
- Комментарии и фото-вложения — основной use case на мобильном (инженер прислал фото с объекта).
- Push-уведомления (PWA уже в стеке) + Telegram-бот как альтернативный канал.
- Дашборды на мобильном — **только KPI-карточки и risk-list**, тяжёлые графики недоступны (сообщение «Откройте на ПК для полной аналитики»).
- Touch targets ≥ 44px, тестируем на iPhone SE (самый узкий экран в активной поддержке).

#### Что одинаково на всех устройствах
- Одна и та же модель данных и API — никакой mobile-only логики на бэке.
- Real-time обновления через WebSocket работают везде.
- Тёмная тема, шрифты, иконки.
- Глубокие ссылки и share — одинаковый формат URL.

### 4.2 Экраны (страницы)

1. **Inbox / My Tasks** — главная для исполнителя. Группировка: Today / This week / Upcoming / No date.
2. **Project view** с табами:
   - **List** — плотный список, как в Linear.
   - **Board (Kanban)** — колонки по статусу, drag-n-drop.
   - **Timeline (Gantt)** — недели/месяцы/кварталы, зависимости стрелками, milestones ромбами.
   - **Calendar** — задачи с датами на месячной сетке.
   - **Files** — все вложения проекта.
3. **Cycle view** — текущий спринт, burndown-чарт сверху, список задач снизу.
4. **Goals** — древовидное представление целей с прогрессом-кольцом.
5. **Dashboard руководителя** (см. §5).
6. **Task detail** — модальное окно или side-panel (Linear-style), не полноэкранная страница.
7. **Map → Tasks** — на странице карты у объекта вкладка «Задачи» (наша уникальная связка).

### 4.3 Визуализация таймлайнов

Главное конкурентное преимущество модуля — **красивый и быстрый Gantt**:

- Библиотека: **`@dhx/trial-react-gantt`** или **`frappe-gantt-react`** (легче, бесплатно) или собственная реализация на **`d3-time-scale + react-flow`**. Рекомендация: на старте `frappe-gantt` (форк), потом — кастомный.
- Зум: день / неделя / месяц / квартал, плавное переключение.
- Сегодняшняя линия (red marker), выходные затенены, milestones ромбами.
- Drag для изменения дат, drag-from-edge для изменения длительности.
- Зависимости рисуются стрелками с правильным offsetting (`react-flow` edges).
- Lazy-render: за пределами viewport задачи не рендерятся.
- Группировка: по сотруднику / по проекту / по области.
- Экспорт в PNG/PDF.

### 4.4 Дашборды для руководителей

**Один дашборд = одна страница, без вкладок.** Виджеты:

- 🔵 **KPI-карточки сверху:** Open Tasks, In Progress, Overdue, Done за период.
- 📊 **Throughput** — линейный график выполненных задач по неделям.
- ⏱ **Lead time / Cycle time** — медиана времени от создания до закрытия.
- 🔥 **Burndown** активного спринта.
- 🗺 **Heatmap по областям** — сколько открытых задач в каждой (с привязкой к нашей карте).
- 👥 **Загрузка по исполнителям** — горизонтальные бары.
- 🚨 **Risk-list** — топ-10 задач с риском срыва (просрочены или близко к due_date и не in_progress).
- 📅 **Mini-Gantt** на 4 недели вперёд по milestones.

Виджеты — `Recharts` (уже в стеке). Опционально drag-n-drop расположение через `react-grid-layout`.

## 5. ТЕХНИЧЕСКИЙ ПЛАН

### 5.1 Backend (FastAPI)

Новый модуль `app/modules/taskops/` со структурой как у существующих:
```
taskops/
  router.py           # /api/v1/taskops/*
  service.py          # бизнес-логика
  permissions.py      # check_project_access, check_external_visibility
  schemas/            # pydantic
  models/             # sqlalchemy
  dependencies.py
```

**Эндпоинты (минимум):**
- `GET/POST /projects`, `GET/PATCH/DELETE /projects/{id}`
- `GET/POST /projects/{id}/tasks`, `PATCH/DELETE /tasks/{id}`
- `POST /tasks/{id}/dependencies`
- `POST /tasks/{id}/comments`, `POST /tasks/{id}/attachments`
- `GET /me/inbox` — мои задачи сгруппированные
- `GET /cycles/{id}` + `POST /projects/{id}/cycles`
- `GET /dashboards/{name}` — агрегированные данные
- `GET /goals`, `POST /goals`
- WebSocket `/ws/taskops` — события `task.updated`, `comment.added`

**Миграция Alembic:** новые таблицы `projects`, `tasks`, `subtasks` (или self-FK), `task_dependencies`, `comments`, `labels`, `task_labels`, `cycles`, `goals`, `project_members`, `attachments`.

**Celery jobs:**
- Автозакрытие cycle и перенос невыполненного.
- Автогенерация инцидент-задач из правил.
- Уведомления о просрочках.
- Push/Telegram нотификации (инфраструктура в проекте уже есть).

### 5.2 Frontend (React)

Новая фича-папка `src/features/taskops/`:
```
taskops/
  api/                # react-query hooks
  components/
    TaskRow.tsx
    TaskBoardColumn.tsx
    TaskDetailPanel.tsx
    GanttChart/
    CommandPalette.tsx
    Dashboard/
      Widgets/
  pages/
    Inbox.tsx
    ProjectPage.tsx   # содержит View-switcher
    CyclePage.tsx
    GoalsPage.tsx
    DashboardPage.tsx
  hooks/
  store/              # zustand: UI-state (открытая панель, текущий view)
```

**Ключевые библиотеки (новые):**
- `@tanstack/react-query` (уже есть) — серверное состояние.
- `@dnd-kit/core` — drag-n-drop в Board и Gantt.
- `frappe-gantt-react` (или fork) — Gantt MVP.
- `cmdk` — command palette ⌘K.
- `react-markdown` + `remark-gfm` — рендер описаний и комментариев.
- `react-grid-layout` — раскладка дашборда (опц., v1.1).
- `recharts` (уже есть) — графики.

### 5.3 Real-time

Используем уже подключённый Socket.IO. События:
- `task:created`, `task:updated`, `task:deleted`
- `comment:created`
- `cycle:closed`
В клиенте — обновление кэша React Query через `queryClient.setQueryData`.

### 5.4 Изоляция аутсорсеров

- Middleware на уровне router'а: для роли `external_dev` все запросы фильтруются по `project_members`.
- Поле `is_external_visible` на задаче — дополнительный фильтр, чтобы внутренние задачи в общем проекте оставались скрытыми.
- Отдельный лендинг при логине: внешний разработчик попадает сразу в свой Inbox без сайдбара со всей админкой.
- Аудит-лог всех действий внешних пользователей (таблица `external_audit_log`).

## 6. ROADMAP (4 фазы, 10–12 недель)

### Фаза 1. MVP (3 недели)
- Модели, миграции, базовый CRUD `/projects`, `/tasks`, `/comments`.
- RBAC + project_members.
- Frontend: Inbox, Project List view, Task detail panel, создание задач.
- Привязка задачи к `location_id`.

## 9. ОТКРЫТЫЕ ВОПРОСЫ К ОБСУЖДЕНИЮ

1. ~~Расширяем существующий `app/modules/tasks/` или создаём `taskops/` рядом?~~ → **Решено: создаём отдельный `taskops/`**. Существующий `tasks/` оставляем для оперативных задач с карты (инциденты), `taskops/` — для полноценного проектного управления. Связь только через `location_id`.
2. ~~Используем t-shirt-оценки или часы (или оба)?~~ → **Решено: оба.** Поле `estimate_type` на уровне проекта: `t_shirt` или `hours`. Руководитель выбирает при создании проекта — и все задачи внутри используют выбранный тип. Внутренние проекты → t-shirt, аутсорс-проекты → часы.
3. ~~Нужны ли Goals на старте или достаточно Projects?~~ → **Решено: Goals — v1.1**, на старте только Projects. Иерархия пока: `Project → Task → Subtask`.
4. ~~Где хранить вложения: локальный volume (есть) или S3-совместимое хранилище?~~ → **Решено: локальный volume** (уже есть в инфраструктуре).
5. ~~Один общий cycle на всю компанию или per-project cycles?~~ → **Решено: per-project cycles.** Аутсорс-разработка — спринты по 2 недели. Проекты внедрения в регионах — этапы с вехами (milestones). Оперативные задачи инженеров — без цикла, просто очередь.
6. Автогенерация инцидент-задач из карты — **отложено на v2.** Celery-job, который по правилам (объект не передаёт данные N дней и т.п.) автоматически создаёт задачу нужному инженеру. Правила, назначение по региону и антидубли — прорабатываются отдельно перед реализацией.
