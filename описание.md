# Концепция и архитектура системы eMedosmotr-map

Данный документ описывает структуру, технологический стек и функциональные возможности системы мониторинга и управления процессами медицинского осмотра.

---

## 1. Архитектура и технологический стек

Выбранный стек технологий обеспечивает масштабируемость, производительность и удобство работы на различных устройствах.

*   **Frontend:** `React` + `TypeScript`. 
    *   Разработка по принципу **Mobile First** или в виде **PWA (Progressive Web App)** для комфортной работы инженеров в полевых условиях (командировках).
*   **Backend:** `FastAPI` (Python). 
    *   Высокая скорость разработки API, строгая типизация (через `Pydantic`) и отличная производительность за счет асинхронности.
*   **База данных:** `PostgreSQL` + расширение `PostGIS`. 
    *   Критически важно для эффективного хранения координат (широта/долгота), работы с полигонами `OpenStreetMap` (границы областей и районов Казахстана) и быстрых пространственных запросов.
*   **Архитектурный паттерн:** `Модульный монолит`.
    *   Планомерный переход к микросервисам. Система разделена на независимые домены (Helpdesk, тендеры, разработка) уже на уровне проектирования БД и API.

## 2. Ключевые модули системы (Фаза 1)

### 2.1. Геоинформационный модуль (Интерактивная карта)
Основа системы — дашборд руководителя с визуализацией данных на карте.

*   **Слои и картография:** Использование подложки `Yandex Maps`. Границы областей (Алматинская, Жетысу и др.) отрисовываются через `GeoJSON` полигоны, загруженные из OSM.
*   **Управление объектами:** Возможность ручного добавления объектов (ЦРБ, военкоматы, частные клиники) с указанием типа, названия и координат.
*   **Визуализация и кластеризация:** 
    *   Использование `Yandex Map ObjectManager` для предотвращения перегрузки карты.
    *   **Масштаб страны:** Отображение агрегированных данных (количество объектов, процент успеваемости региона).
    *   **Детализация:** При приближении маркеры разделяются на конкретные учреждения.
    *   **Цветовая индикация:** 
        *   🟢 **Зеленый:** анализы прикрепляются исправно.
        *   🔴 **Красный:** проблема (призывники уходят в сторонние клиники).

### 2.2. Модуль аналитики и интеграции с eMedosmotr
Ядро системы для контроля ввода данных и минимизации потерь.

*   **Интеграция:** Однонаправленная синхронизация (ETL-процесс или чтение из `Read-Replica` базы eMedosmotr). Позволяет забирать срезы данных без нагрузки на основную систему.
*   **Метрики:** Визуализация воронки: *Количество выданных направлений ВВК -> Количество загруженных результатов*. Система автоматически подсвечивает районы с низкой конверсией.
*   **Учет оборудования:** Отображение статуса подключения мед. оборудования (автоматическая передача через сервер-ноутбук или ручная загрузка).

### 2.3. Модуль управления задачами и персоналом
*   **Постановка задач:** Привязка задач к конкретным объектам на карте с назначением ответственных региональных инженеров.
*   **KPI и тайм-трекинг:** Сводки по объему выполненных работ и закрытых задач. Инженеры могут обновлять статус задач со смартфонов непосредственно на объекте.

## 3. Развитие системы (Фаза 2)

### 3.1. Helpdesk (Техническая поддержка)
*   Многоуровневая система тикетов:
    *   **1-я линия:** операторы.
    *   **2-я линия:** региональные инженеры.
    *   **3-я линия:** разработчики.
*   Привязка обращений к объектам на карте и конкретным врачам.

### 3.2. Трекинг договоров и тендеров (CRM-блок)
*   Ведение карточек регионов и управлений мобилизационной подготовки.
*   **Pipeline статусов:** Планирование закупа → Тендер → Договор → Внедрение → Подписание актов.
*   Автоматические уведомления о сроках окончания техподдержки.

### 3.3. Модуль контроля подрядчиков (Разработка)
*   Изолированное рабочее пространство для системных аналитиков.
*   Постановка задач внешним разработчикам (аутсорс).
*   Трекинг релизов, контроль трудозатрат и приемка работ по eMedosmotr.

### 3.4. Внутренние коммуникации
*   Встроенный защищенный чат с ролевой моделью доступа.
*   Контекстные обсуждения: возможность создавать чаты внутри карточки объекта или задачи для сохранения истории общения.

---

## Итог
Данный подход обеспечивает прозрачный контроль работы инженеров «в поле» и предоставляет руководству инструмент для аргументированного диалога с государственными органами на основе реальной аналитики.
